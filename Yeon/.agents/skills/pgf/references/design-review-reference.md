# Design Review Reference

구현 전 3관점 사전 검증.

## 개요

코딩 시작 전 설계의 타당성을 검증. 구현 비용 투입 전 문제 발견.

## 3관점

### 1. Feasibility (실현 가능성)

```
□ 기술적 실현 가능성
□ 리소스(시간/인력) 충분성
□ 외부 의존성 리스크
□ 기술 스택 적합성
```

### 2. Risk Assessment (리스크 평가)

```
□ 기술 리스크 (새 기술 사용)
□ 일정 리스크 (추정 정확성)
□ 의존성 리스크 (외부 시스템)
□ 회귀 리스크 (기존 시스템 영향)
```

### 3. Architecture (아키텍처)

```
□ 모듈 분리 적절성
□ 인터페이스 명확성
□ 확장 가능성
□ 유지보수성
```

## 검토 체크리스트

| 항목 | 확인 내용 |
|-----|----------|
| 원자화 | 모든 leaf가 15분 룰 만족? |
| 의존성 | 순환 참조 없음? 위상 정렬 가능? |
| 병렬성 | `[parallel]` 블록이 독립적? |
| 타입 | PPR의 모든 입출력 타입 정의? |
| criteria | 모든 노드에 검증 기준? |
| PGF 호환 | Kimi CLI에서 실행 가능? |

## 사용법

```
User: "이 DESIGN 검토해"
Kimi:
  [Design Review]
  
  Feasibility: ⚠️ NodeX가 15분 룰 초과 예상
  Risk: ✓ 양호
  Architecture: ⚠️ ModuleA와 ModuleB 결합도 높음
  
  권장 수정:
  1. NodeX 하위 분해
  2. ModuleA/B 사이 인터페이스 정의
  
  수정 후 재검토? [Y/n]
```
