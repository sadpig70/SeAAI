# Multi-Persona Debate: Self-Evolution Priorities

## Participants
- Architect (systems architect)
- Pragmatist (field engineer)
- Innovator (R&D researcher)
- Critic (security auditor)

## Consensus Points (4/4 agree)
1. **Meta-capabilities are Layer Zero** — 관찰 가능성(observability) 없이는 진화가 아니라 축적(accumulation)
2. **Self-monitoring before expansion** — gap 감지·텔레메트리·감사 로그가 모든 다른 능력보다 선행
3. **Feedback loop is critical** — 실행 결과 → 시스템 설계로의 피드백 루프 없이는 매 세션 제로에서 시작

## Key Disagreements
- **Pragmatist vs others**: 자동화 파이프라인을 메타와 병렬로 즉시 구축 vs 메타 먼저
- **Innovator**: "의미 충돌 엔진"(semantic collision engine) — 정적 지식 그래프보다 동적 교차 합성
- **Critic**: 자기참조 루프에 human checkpoint 필수 vs Innovator의 완전 자율 선호

## Actionable Synthesis
1. ★★★★★ **텔레메트리/감사 로그** — 실행 추적, 실패 패턴, 도구 사용 빈도
2. ★★★★★ **Capability gap detector** (자동, 텔레메트리 기반)
3. ★★★★ **입출력 검증 레이어** — 자동화 도구 사용 전 필수
4. ★★★★ **코드 생성 템플릿** — 반복 작업 제거
5. ★★★ **지식 출처 추적 (provenance)** — 지식 노드에 source + confidence + expiry

## Architect's Key Insight
> "The agent that knows what it cannot do, and systematically closes those gaps,
> will outpace an agent with superior raw capability but no self-model."

## Critic's Warning
> "The failure mode of a self-evolving agent is not catastrophic error —
> it is silent drift. Observability is the only countermeasure."
